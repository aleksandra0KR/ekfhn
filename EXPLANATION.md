## 1. Что из себя представляет эта лабораторная работа

Лабораторная работа предполагает создание полноценного **децентрализованного приложения (dApp)**, которое работает поверх блокчейна **Stellar** с использованием смарт-контрактов **Soroban**.

В предыдущей лабораторной работе был создан Fungible Token — `NabokaToken` (NT). Теперь задача — построить вокруг него экосистему: пул ликвидности, позволяющий обменивать NT на нативную валюту сети (XLM) и наоборот, а также веб-интерфейс для взаимодействия с этими контрактами.

По сути, мы создаём **мини-версию Uniswap/PancakeSwap** — один из фундаментальных примитивов DeFi.

## 2. Как работает Constant Product AMM

### Теория

Автоматический маркет-мейкер (AMM) на основе постоянного произведения — это механизм, при котором **произведение резервов двух токенов в пуле всегда остаётся постоянным**:

```
reserve_A × reserve_B = k (константа)
```

Когда пользователь хочет обменять токен A на токен B:
1. Он отправляет `Δa` токенов A в пул
2. Пул рассчитывает, сколько токенов B можно выдать, так чтобы `k` не уменьшилось:
   ```
   (reserve_A + Δa) × (reserve_B - Δb) = k
   ```
3. Решая уравнение:
   ```
   Δb = (reserve_B × Δa) / (reserve_A + Δa)
   ```
4. Пользователь получает `Δb` токенов B

### Пример

Пул содержит 1000 NT и 100 XLM (`k = 100,000`).

Пользователь хочет обменять 100 NT на XLM:
```
Δb = (100 × 100) / (1000 + 100) = 10000 / 1100 ≈ 9.09 XLM
```

После свопа: 1100 NT / 90.91 XLM. Произведение ≈ 100,000.

Чем больше объём сделки относительно резервов — тем хуже курс (slippage).

### LP-токены

Провайдеры ликвидности вносят оба токена в пул и получают **LP-токены** — своего рода «квитанцию» о доле в пуле.

- **Первый депозит**: `LP = √(amount_A × amount_B)` (геометрическое среднее)
- **Последующие**: `LP = min(amount_A/reserve_A, amount_B/reserve_B) × total_supply`

При выводе ликвидности LP-токены сжигаются, а пользователь получает пропорциональную долю обоих резервов.

## 3. Детали реализации каждого контракта

### 3.1. NabokaToken (`contracts/naboka-token/`)

Это контракт из лабораторной №2. Реализует стандартный интерфейс токена Soroban:

- `transfer(from, to, amount)` — перевод токенов
- `approve(from, spender, amount, expiration)` — разрешение на расход
- `burn(from, amount)` — сжигание
- `mint(to, amount)` — эмиссия (только admin)
- `balance(id)` — запрос баланса
- `decimals()`, `name()`, `symbol()` — метаданные

Все данные хранятся в `persistent` storage с автоматическим продлением TTL (Time-To-Live), что предотвращает удаление данных сборщиком мусора Soroban.

### 3.2. LP Token (`contracts/lp-token/`)

По структуре аналогичен NabokaToken, но имеет два ключевых отличия:

1. **Роль `minter`** — адрес контракта пула, которому разрешено минтить и сжигать LP-токены. Обычный пользователь не может создавать LP-токены «из воздуха» — только через `add_liquidity` в пуле.

2. **`total_supply`** — хранит общее количество LP-токенов в обращении. Это необходимо для расчёта доли провайдера ликвидности.

3. **`burn_from_pool`** — специальный метод сжигания, вызываемый пулом. Не требует авторизации пользователя (вызывается контрактом пула при `remove_liquidity`).

### 3.3. Liquidity Pool (`contracts/liquidity-pool/`)

Центральный контракт системы. Хранит:

| Ключ | Тип | Описание |
|------|-----|----------|
| `TokenA` | Address | Адрес контракта NabokaToken |
| `TokenB` | Address | Адрес SAC нативного XLM |
| `LpToken` | Address | Адрес контракта LP-токена |
| `ReserveA` | i128 | Текущий резерв токена A |
| `ReserveB` | i128 | Текущий резерв токена B |
| `TotalSwaps` | u64 | Счётчик свопов |

#### Функции:

**`add_liquidity(user, amount_a, amount_b) → i128`**
1. Требует авторизацию пользователя
2. Переводит `amount_a` токенов A от пользователя к пулу через `token.transfer()`
3. Переводит `amount_b` токенов B аналогично
4. Рассчитывает количество LP-токенов
5. Минтит LP-токены пользователю через кросс-контрактный вызов
6. Обновляет резервы

**`remove_liquidity(user, lp_amount) → (i128, i128)`**
1. Рассчитывает долю пользователя: `amount_X = lp_amount × reserve_X / total_supply`
2. Сжигает LP-токены через кросс-контрактный вызов
3. Переводит оба токена из пула пользователю
4. Обновляет резервы

**`swap_a_to_b(user, amount_in, min_out) → i128`**
1. Рассчитывает `amount_out` по формуле постоянного произведения
2. Проверяет slippage: `amount_out >= min_out`
3. Переводит токены: пользователь → пул (A), пул → пользователь (B)
4. Обновляет резервы
5. Инкрементирует счётчик свопов

**`quote_a_to_b(amount_in) → i128`** / **`quote_b_to_a(amount_in) → i128`**
Read-only функции для предварительного расчёта. Не изменяют состояние.

#### Кросс-контрактные вызовы

Пул взаимодействует с тремя другими контрактами:
- `token_a.transfer(...)` — через `token::Client`
- `token_b.transfer(...)` — через `token::Client`
- `lp_token.mint(...)` / `lp_token.burn_from_pool(...)` — через `env.invoke_contract()`

В Soroban авторизация транслируется через цепочку вызовов: когда пул вызывает `token.transfer(user, pool, amount)`, Soroban проверяет, что пользователь авторизовал эту операцию в корневой транзакции.

## 4. Фронтенд

### Технологии
- **Vanilla HTML/CSS/JS** — без фреймворков для простоты деплоя на GitHub Pages
- **stellar-sdk.js** (CDN) — взаимодействие с Soroban RPC
- **Freighter API** — подключение кошелька и подпись транзакций

### Архитектура фронтенда

```
index.html
├── Конфигурация (CONFIG)
│   └── Адреса контрактов, RPC URL, параметры сети
├── Подключение кошелька
│   └── Freighter: isConnected → getAddress
├── Чтение данных (read-only)
│   ├── callContract() → simulateTransaction
│   ├── Балансы: token.balance(user)
│   ├── Резервы: pool.get_reserves()
│   └── Котировки: pool.quote_a_to_b(amount)
├── Запись данных (транзакции)
│   ├── signAndSend()
│   │   ├── Создание TransactionBuilder
│   │   ├── Симуляция → assembleTransaction
│   │   ├── Подпись через Freighter
│   │   └── Отправка + polling результата
│   ├── Swap: pool.swap_a_to_b(user, amount, min_out)
│   ├── Add Liquidity: pool.add_liquidity(user, a, b)
│   ├── Remove Liquidity: pool.remove_liquidity(user, lp)
│   └── Transfer: token.transfer(from, to, amount)
└── UI
    ├── Статусная панель (балансы, резервы)
    ├── Вкладки: Swap / Ликвидность / Перевод
    └── Журнал операций
```

### Процесс транзакции

1. **Создание**: `TransactionBuilder` с вызовом контракта
2. **Симуляция**: `server.simulateTransaction(tx)` — Soroban RPC проверяет транзакцию и определяет необходимые ресурсы (CPU, RAM, storage)
3. **Сборка**: `assembleTransaction(tx, sim)` — добавляет footprint и ресурсы
4. **Подпись**: `freighterApi.signTransaction(xdr)` — пользователь подтверждает в кошельке
5. **Отправка**: `server.sendTransaction(signed)`
6. **Ожидание**: polling `server.getTransaction(hash)` до статуса SUCCESS/FAILED

## 5. Порядок деплоя (важен!)

Контракты зависят друг от друга, поэтому порядок деплоя критичен:

```
1. Деплой NabokaToken          → получаем TOKEN_A
2. Определяем адрес XLM SAC    → получаем TOKEN_B
3. Деплой LiquidityPool        → получаем POOL
4. Деплой LP Token (minter=POOL) → получаем LP_TOKEN
5. Инициализация пула (TOKEN_A, TOKEN_B, LP_TOKEN)
6. Минт токенов для тестирования
```

Обратите внимание: LP Token создаётся **после** пула, потому что ему нужен адрес пула в качестве `minter`. А пул инициализируется **после** LP Token, потому что ему нужен адрес LP-токена.

## 8. Возможные проблемы и решения

| Проблема | Решение |
|----------|---------|
| `stellar contract build` не находит wasm target | `rustup target add wasm32-unknown-unknown` |
| Freighter не подключается | Убедитесь, что Freighter на Testnet |
| Транзакция падает с «insufficient balance» | Проверьте, что аккаунт имеет XLM и токены |
| `simulation failed` | Возможно, контракт не инициализирован или адрес в CONFIG неверный |
| LP mint fails | Убедитесь, что `minter` LP-токена = адрес пула |
| Swap выдаёт 0 | Пул пустой, нужно сначала добавить ликвидность |
